Parallel Programming in Python with celery

Date Sat 13 December 2014 Tags python / linux

Parallel Programming

This is the second note on parallel programming. Please read Parallel Programming with IPython"

Celery

Celery offers a different approach to setting up the parallel computation, and is much more restrictive than IPython. Celery requires that any function that can be run in parallel must be decorated, and loaded by the worker node. Thus, the code must already reside on the worker nodes computer. This avoids the security issue of arbitrary python code being trivially run.

Celery also uses different backend services to transport messages between the client, workers and controllers. My favourite feature of celery is the cluster status monitor flower, which sets up an interactive website on localhost.

Dependencies

Celery has much more complex dependencies than IPython, look at the article first steps with python for a comprehensive list of the options. I'm merely going to demonstrate what I did to get it working as I followed that guide.

Message Queue

I chose the most simple one, RabbitMQ. Do this:

$ sudo apt-get install rabbitmq-server

I have since had some issues with RabbitMQ, in particular, I find it very hard to delete messages which I might accidentally setup. To remove these messages I need to:

  1. shutdown rabbitmq using rabbitmq-ctrl script
  2. delete all folders in /usr/share/rabbitmq/mnesia/
  3. restart rabbitmq using /etc/init.d/rabbitmq restart

There must be a better way!

Defining the tasks

Assuming celery is installed using pip, we can now begin coding. We first create a configuration object, and then decorate functions with that object. These functions form a list of functions which workers can execute. Save the following script as tasks.py

from celery import Celery
app = Celery('tasks', 
              broker='amqp://guest@localhost//',
              backend='amqp')

import numpy as np

@app.task
def rand_det(n): 
    A = np.random.rand(n,n)
    A = A.T.dot(A)
    return np.linalg.det(A)

Observe that the decorator @app.task must be used for every function that you intend to distribute. There are two terms which need defining:

  • broker : this is the message passing backend which will transmit the jobs to and from the worker nodes and the client.
  • backend : this is some form of database which will contain the final results of the runs.

There are several choices for broker and backend, but this article is using RabbitMQ for both.

Running celery

Now that we have defined the jobs we can start a cluster to perform these jobs. This is done with the following shell command:

$ celery -A tasks worker --loglevel=info

The -A option specifies the module to load to look for tasks, while the worker option tells celery to start a set of workers.

If you have installed flower, you can also start it with:

$ celery flower -A tasks

This will create a website at localhost:8000, its well worth a look.

Running a job

We can now run jobs, the easiest way to do this is to load up a python shell. 

import tasks
handle = tasks.rand_det.apply_async(10)
print(handle.get())

This will use the celery cluster to run the function once with the input parameter 10. Calling the apply_async method prevents the current script from waiting for the result, while the call to handle.get() will halt the program until a result is returned.

Setting up multiple computers.

Now lets make celery easy for multiple computers. We need to make a few assumptions. The computer running rabbitMQ for brokering and result back-end must have a known IP or computer name, and it must be visible from all computers, further, let the connection between the workers and this server be secure so we do not need to worry about encryption. Lets use 192.168.1.2. We can now write a configuration script, that can be given to all the workers and clients, that describes this topology. Call it celeryconfig.py:

# To set these, change the environment variables instead.
CELERY_RESULT_BACKEND = 'amqp://192.168.1.2:5672'
BROKER_URL = 'amqp://192.168.1.2:5672'

CELERY_TASK_SERIALIZER = 'pickle'
CELERY_RESULT_SERIALIZER = 'json'
CELERY_ACCEPT_CONTENT=['pickle', 'json']
CELERY_TIMEZONE = 'Australia/Sydney'
CELERY_ENABLE_UTC = True
CELERYD_MAX_TASKS_PER_CHILD=1

The usage is now the same from the clients perspective, but now any number of workers can connect and disconnect and the server will seamlessly handle it. Note the setting CELERYD_MAX_TASKS_PER_CHILD=1 ensures that the python interpreter is restarted after each job. For simple jobs such as this one its not necessary and will slow things down, however long running computation jobs will probably not notice the performance hit.

To join the cluster, simply install the code onto the new computer and type:

celery worker -A tasks --config=celeryconfig

The flower dashboard will also update to reflect the new worker.

Comparison between Celery and IPython

Both Celery and IPython.Parallel achieve the same purpose. In my opinion IPython is best when you have one single large computer. Although celery is harder to use, the ease with which computers can join the cluster and the usefulness of the flower dashboard make it my go to choice if I want to build a large python cluster.
